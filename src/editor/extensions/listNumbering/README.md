# Ordered-list numbering engine

Custom multilevel numbering for ordered lists ‚Äî presets (`1. a. i.`, legal
`1. 1.1. 1.2.1.`, ‚Ä¶) and per-level customization (style, separator, start-at,
parent-inclusion) with composite markers like `1.a` / `1.a.i` that native
Tiptap/ProseMirror `<ol>`s cannot produce.

## Why it's custom

Tiptap ordered lists render a plain `<ol>` and persist only `start`. They can't
store per-level type or render composite markers, and CSS-only styling
(`ol ol { list-style-type: ‚Ä¶ }`) isn't serialized, so it doesn't round-trip.
So the numbering **definition lives in the document model** and markers are
drawn with **CSS counters** driven by that definition.

## Data model

```ts
type NumberStyle = 'decimal' | 'decimalZero' | 'lowerAlpha' | 'upperAlpha' | 'lowerRoman' | 'upperRoman';
type Separator   = 'dot' | 'paren' | 'parens';                 // a.  a)  (a)
interface ListLevelConfig { style; separator; startAt; includeParent }
type ListDefinition = ListLevelConfig[];                        // index = level, up to 9
```

**Storage ‚Äî a document-level registry (Word `abstractNum`-style).**
- The **Document node** carries a `listDefs` attribute: `{ [id]: ListDefinition }`.
- Each **ordered-list node** carries `listDefId` pointing into the registry
  (plus native `start` and a `restart` flag).
- A list node's **level = its ordered-list nesting depth** (ordered ancestors
  only) ‚Äî computed at render time, never stored ‚Äî so a list nested inside a
  bullet list still numbers by ordered depth.

Because the configured persistence format is **ProseMirror JSON**
(`editor.getJSON()`), both the registry (a doc attr) and the per-list id (a node
attr) serialize and re-hydrate for free. Registry writes go through
`tr.setDocAttribute('listDefs', ‚Ä¶)`, so they're **undoable** and one transaction.
Definitions are deduped by content hash (`definitionId`), so identical
lists share one registry entry; a per-level edit **copy-on-writes** a new id and
re-points only the edited list.

## Rendering ‚Äî CSS counters, one counter per depth

`counterCss.ts` generates scoped CSS per definition. The trick: **one counter
per depth level** (`pgnol1..pgnol9`), reset on each `<ol>` of that level and
incremented on its `<li>`s. Counter values inherit to descendants, so a nested
level's `::before` references its ancestors' counters. The marker is composed
from individual `counter(name, style)` pieces ‚Äî which is what lets each level use
a **different** style (`1.a`), something a single `counters()` can't express.
Rules are keyed `.docs-page-content ol[data-list-def="ID"][data-list-level="d"]`.

The plugin (`extension.ts`) stamps every `<ol>` with `data-list-def` (inherited
from the nearest assigned ancestor, so lists indented into deeper levels inherit
the definition) and `data-list-level` via **node decorations** ‚Äî no doc/history
churn ‚Äî and rebuilds a `<style>` element whenever the registry changes.

`NumberStyle ‚Üí CSS`: decimal, `decimal-leading-zero`, `lower/upper-alpha`,
`lower/upper-roman`.

## Presets

Six, in `model.ts` (`PRESETS`): `decimal` (1. a. i.), `paren` (1) a) i)),
`legal` (1. 1.1. 1.2.1. ‚Äî decimal + parent-inclusive at every level),
`upperAlpha` (A. a. i.), `upperRoman` (I. A. i.), `zero` (01. a. i.). Selecting
one overwrites all levels. Levels beyond 3 are generated by `extendDefinition`
(cycling the last three), so deep nesting is defined.

### Adding a preset

Add an entry to `PRESETS` with `id` + a `levels: ListDefinition`. Then add a
matching decorative card to `PRESET_CARDS` in
`components/NumberedListStylePicker.tsx` (same `id`) with its 4 miniature
marker strings. Nothing else to wire ‚Äî the card applies via `applyListPreset`.

## Commands (all no-ops ‚Üí `can()` false outside an ordered list)

`applyListPreset(id)`, `applyListDefinition(def)` (whole draft at once),
`setLevelNumberStyle/Separator/StartAt/IncludeParent(level, ‚Ä¶)`,
`addListLevel()`, `resetListLevel(level)`, `restartNumbering()`.
`getActiveListInfo(editor)` reads the list at the cursor (level, definition,
matched preset) for the picker/panel.

## UI wiring

`NumberedListMenu` (in the toolbar's list group) hosts the preset popover and
the customize dialog. The dialog edits a **local draft** ‚Äî the live preview uses
the same `renderMarker` the engine renders with ‚Äî and only commits on **Apply**
(`applyListDefinition`, one undoable step). Cancel discards; Reset level restores
that level to its default.

## Persistence & interop

- **JSON (primary):** full round-trip. Verified by test ‚Äî registry on the doc
  attr survives `getJSON()` ‚Üí reload.
- **HTML export:** each `<ol>` emits `data-list-def-id`; the registry itself
  rides in JSON. Standalone-HTML fidelity of the *definition* (for external
  paste into another app) is **not** carried ‚Äî a documented limitation. Paste
  within the same document works (the registry entry already exists).
- **DOCX:** `docxNumbering.ts` maps a definition ‚Üí Word numbering
  (`abstractNum`/`num`): `numFmt`, `lvlText` templates (`%1.`, `%1.%2` for
  composites), `start`. CSS `::before` markers never export to Word, so a DOCX
  exporter must read **this** mapping. The mapping is library-agnostic (plain
  OOXML-shaped data); feed `toDocxNumbering(registry)` into e.g. the `docx`
  npm package's numbering config. A full whole-document DOCX exporter (tables,
  redlines, ‚Ä¶) does not exist in the repo and is out of scope ‚Äî this is the
  numbering piece, ready to plug in.
- **Print / pagination:** the print renderer clones each top-level block onto
  its page, so a whole `<ol>` stays on one page (Bar A) ‚Äî counters never reset
  mid-list. The generated `<style>` and the decoration attributes are carried
  into the clones, so composite markers print correctly (verified). If the
  renderer is ever changed to split a list across pages, continued fragments
  would need `counter-set` offset injection.

## Deferred

**Continue numbering across separate lists** (list ‚Üí paragraph ‚Üí list). CSS
counters restart per `<ol>` by default; auto-continue needs computed `start`
offsets. Native `start`/`restart` still work; `restartNumbering()` sets the
restart intent (and, since lists already restart by default, is currently a
visual no-op ‚Äî it's the counterpart to the deferred continue mode).
